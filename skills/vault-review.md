# /vault-review

Review accumulated system improvement proposals and triage vault inbox.

## Process

1. Read system-health-board.md for backlog items
2. Read ~/vault/inbox/ for uncategorized items
3. Read declined-proposals.md for context on what was already rejected
4. Present each pending item for user decision: implement, defer, decline
5. Update boards and files based on decisions

## Vault Inbox Triage

6. Scan ~/vault/inbox/ for notes with `status: inbox` in frontmatter
7. Present each inbox note to user for categorization:
   - Move to ~/vault/claims/ — factual claims, assertions, hypotheses
   - Move to ~/vault/frameworks/ — mental models, processes, methodologies
   - Move to ~/vault/sources/ — reference material, external documents
   - Delete — not worth keeping
8. **Conflict detection**: Before promoting, search existing vault notes via qmd for related content (by title keywords and tags). If related notes are found:
   - Present both the inbox note and the existing note(s) to the user
   - Ask: is this new knowledge, or does it supersede an existing note?
   - If superseding: mark the old note `status: superseded` with `superseded_by: [[new note title]]`
9. Update the note's `status` field from `inbox` to `active` (or the chosen category)
10. Add [[wikilinks]] to related vault notes during triage

## Note Lifecycle

Vault notes follow a status progression:

```
inbox → active → superseded
```

- `inbox` — raw/draft, awaiting triage (auto-generated by session-extract hook or brain-ingest)
- `active` — reviewed and accepted as current knowledge
- `superseded` — replaced by a newer note. Frontmatter: `superseded_by: [[new note title]]`
